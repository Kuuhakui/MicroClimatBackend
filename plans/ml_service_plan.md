
# ML Prediction Service - План Обучения и Архитектуры

## 1. Стратегия Обучения (Training Strategy)

Обучение моделей будет разделено на два этапа: базовое обучение и адаптивное дообучение.

### Типы Обучения:
1.  **Базовая Модель (General Model)**: 
    *   Обучается на агрегированных исторических данных всех помещений.
    *   Цель: Уловить общие физические закономерности (нагрев, охлаждение).
    *   Периодичность: Раз в месяц или при значительных изменениях в ПО.
2.  **Специфичные Модели (Room-Specific Models)**:
    *   Дообучение базовой модели на данных конкретного помещения.
    *   Учитывает уникальные параметры (объем, теплопроводность, расположение датчиков).
    *   Периодичность: Раз в неделю или при накоплении достаточного объема новых данных.

### Триггеры Обучения:
*   **Расписание (Scheduled)**: Периодический запуск через Celery/Redis.
*   **Ручной запуск (Manual)**: Через панель администратора во фронтенде.
*   **Дрифт данных (Data Drift)**: Автоматический запуск, если ошибка предсказания (MSE/MAE) превышает заданный порог в течение X часов.

## 2. Архитектура и Типы Данных

### Соображения по Ресурсам (Hardware Considerations):
Учитывая ограниченные аппаратные ресурсы, на начальном этапе будут приоритетны модели с меньшими вычислительными требованиями.
*   **Предпочтительно**: Scikit-learn, XGBoost, CatBoost.
*   **В перспективе**: При наличии достаточных ресурсов возможен переход на PyTorch/TensorFlow для более сложных моделей (LSTM, Transformer).

### Архитектура Модели:
*   **Основной стек**: Python, Scikit-learn, XGBoost/CatBoost (для быстрого развертывания и меньших требований к CPU/RAM).
*   **Тип модели**: 
    *   *Baseline*: Gradient Boosting (XGBoost/CatBoost) на скользящем окне.
    *   *Target (в будущем)*: LSTM (Long Short-Term Memory) или Transformer для работы с временными рядами при наличии достаточных ресурсов.

### Формат Данных:
*   **Источники данных для обучения**: Основным источником исторических данных для обучения будет `data-storage-service` (PostgreSQL), который хранит `sensor_measurements` и `room_physical_params`.
*   **Входные признаки (Features)**:
    *   `temperature_history`: Вектор температур за последние 24 часа (из `data-storage-service`).
    *   `humidity_history`: Вектор влажности за последние 24 часа (из `data-storage-service`).
    *   `room_params`: Объем, площадь, теплопроводность (из `room-management-service`, через `data-storage-service`).
    *   `time_features`: Час дня, день недели (sin/cos encoding).
    *   `external_weather`: (Опционально) Данные с внешних API метеослужб (интеграция будет проработана отдельно).
*   **Выходные данные (Target)**:
    *   Прогноз температуры на следующие 1, 3, 6 часов.
    *   Интервал уверенности (Confidence Interval) — 95%.

## 3. Передача Данных и Форматы

### Типы Соединений:
1.  **Inference (Предсказание)**:
    *   *Протокол*: **Redis Pub/Sub**.
    *   *Процесс*: `sensor-ingestion-service` публикует данные -> `ml-prediction-service` считывает, делает предсказание -> Публикует результат в топик `prediction_results`.
2.  **Training (Обучение)**:
    *   *Протокол*: **gRPC** или **HTTP/REST**.
    *   *Процесс*: Запрос к `data-storage-service` на получение батча исторических данных в формате Parquet или JSON.
3.  **Storage (Хранение моделей)**:
    *   *Протокол*: **S3 API** (через `minio-file-service`).
    *   *Формат*: Сохраненные веса модели (.pth, .onnx, .pkl).

### Формат Сообщений (Inference Payload):
Используется **JSON** для гибкости.

```json
{
  "request_id": "uuid-v4",
  "room_id": "uuid-room-123",
  "timestamp": "2026-02-26T14:00:00Z",
  "current_features": {
    "temp": 22.5,
    "humidity": 45.0,
    "sensors": ["sensor-1", "sensor-2"]
  },
  "predictions": [
    {"time_offset": "1h", "value": 23.1, "conf_low": 22.8, "conf_high": 23.4},
    {"time_offset": "3h", "value": 24.0, "conf_low": 23.5, "conf_high": 24.5}
  ]
}
```

## 4. Метрики и Контроль Качества

*   **RMSE (Root Mean Square Error)**: Основная метрика точности.
*   **Latency**: Время выполнения одного предсказания не должно превышать 100мс.
*   **A/B Тестирование**: Возможность запуска двух моделей одновременно для сравнения точности на реальных данных.
